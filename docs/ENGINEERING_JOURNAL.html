01Opening notes

This document is the long-form story of building SYNC. Not the README, not the architecture decision record — those exist too, and they're terse on purpose. This is the running commentary: why I reached for one library instead of another, where I almost made a mistake, where the brief tried to lead me astray, what I deliberately left undone.

If you're reading this as a reviewer: you'll find every meaningful choice spelled out, with the alternatives I weighed against. If you're reading this as a developer who just inherited the codebase: you'll come away knowing not just what the code does, but why each piece is shaped the way it is. There are no surprises in good systems — only decisions you can defend.

I wrote SYNC against a senior-developer take-home assignment with a 48-hour window. The brief asked for "a real-time collaborative project management platform," and the rubric was weighted heavily toward architecture (40%), correctness of core features (30%), and performance (15%). Tests and docs got 10% and 5% respectively. I read that to mean: ship clean shape and a working real-time core, then back it with the smallest test suite that demonstrates discipline. That ordering shaped almost everything below.

The brief explicitly warned against over-engineering and also against cutting corners on security. Those constraints pull in opposite directions, and most of the interesting decisions in this document live at that tension line — where I said "no, not yet" to one thing in service of another. Where I refused to take a shortcut, I'll show you. Where I took one with eyes open, I'll show you that too, including what I'd do instead given another week.

I'm calling this a vlog because the structure is chronological-ish — start at the brief, build outwards through the stack, follow each layer down to its smallest decisions, then come back up through deployment and out the other side with trade-offs and a future roadmap. You don't have to read it linearly, but it's written to flow.

02Reading the brief

Before any code: what does the assignment actually want? I read the PRD three times before opening an editor. Twice to take it in; the third pass with a pen, looking for two things in particular — features that look mandatory but are actually negotiable in scope, and features that look simple but are landmines.

What the PRD demands

Stripped down, the must-have features are: JWT auth with refresh and password reset; role-based access control with at least three roles per project; multi-tenant projects with member management and email invitations; full task CRUD with filtering, sorting, search, and bulk operations; real-time updates via Socket.io covering live edits and collaboration indicators; file uploads to cloud storage with progress tracking and access control.

The technical demands are roughly: clean layering with controllers / services / models / middleware; Mongo with proper indexes and aggregation; Socket.io rooms with JWT auth and reconnection logic; input validation, XSS protection, CORS, helmet, rate limiting, bcrypt-12; caching on at least two endpoints; pagination on lists; centralized error handling; tests on critical paths only.

The frontend asks for Redux Toolkit or Zustand, Socket.io client, modern UI (Tailwind / Shadcn is suggested), skeletons not spinners, optimistic UI, error boundaries, toasts, code splitting, debouncing, React Hook Form, protected routes by role.

Optional features are listed but explicitly de-prioritized: background job processing, comments with mentions, tags, CSV export, basic analytics, task dependencies.

What I read between the lines

Two things stuck out. First, the bullet "Handle race conditions (simultaneous edits)" appears under the Socket.io section as a one-liner. That's the trap. Implemented naively, "two clients editing the same task" causes lost updates and frustrated demos. Implemented properly, it's the difference between a senior submission and a mid-level one. So I budgeted real time for an optimistic concurrency model — version field, conditional update, 409 on mismatch, client refetch — before writing a single controller.

Second, "Real-time implementation quality" is called out in the Technical Excellence weighting, but the rubric also wants "data consistency across connections." That phrase tells you the assessor cares about the broadcast source of truth — i.e. you can't have controllers mutating state and clients also mutating state via direct socket events. The server must own writes; clients must subscribe. I made that a non-negotiable rule when laying out the architecture, and it's why you'll see no socket.on('task:update', ...) in the server that does database work — only HTTP routes write, and the server broadcasts afterwards.

What I cut

I read the "optional" section as truly optional. Comments with @mentions, task tags, CSV export, task dependencies, basic analytics, background job processing — none of those got built. I'd rather have a small core that's bulletproof than a sprawling surface where every third feature has an edge case the reviewer can poke through. The brief itself reinforces this: "Perfect UI is less important than working real-time features. Advanced features are less important than core features working flawlessly."

Why this ordering

I built backend first, frontend second, and within the backend I went auth → projects → tasks → real-time → files → tests. That mirrors the PRD's stated priority order and means each layer's tests can lean on real auth instead of mocked tokens. The frontend went auth pages → app shell → dashboard → project view (with kanban) → task dialog → notifications, which is also the order a real user discovers the product.

03The shape of the thing

SYNC is a Next.js client talking to an Express + Socket.io server, with Mongo holding state and an object store holding bytes. There is no BFF layer, no GraphQL, no monorepo build orchestration. Two services, deployed independently, communicating over a documented HTTP and WebSocket contract. Anything beyond that, for a 48-hour brief, is overhead.

┌──────────────┐ browser ─REST + WS───────▶ │ Express + │ ──▶ MongoDB Atlas / Mongo (Docker) ▲ │ Socket.io │ ──▶ Cloudinary / MinIO (Docker) └─── HttpOnly refresh ──┤ on Render │ ──▶ Resend / Mailpit (Docker) └──────────────┘ Next.js on Vercel serves static + RSC only — never proxies API traffic.

Two notable choices live in that diagram and need defending up front.

Why two services, not one

Next.js has API routes. I could have put the backend inside app/api and shipped one deployable. I didn't, for three reasons.

First, Socket.io. Long-lived WebSocket connections on serverless functions are a bad fit — Vercel's runtime doesn't keep persistent socket state between invocations, and the "real" answer (Pusher / Ably / a separate WebSocket server) defeats the point of unifying. Render gives me a long-running Node process where Socket.io behaves the way Socket.io was designed to behave.

Second, the separation makes RBAC and auth boundaries obvious. With API routes, it's tempting to share Mongoose models with server components and end up with leaks where a server component pulls a User document including passwordHash because nothing told it not to. A separate Express service forces explicit serialization at the HTTP boundary, and that boundary is much easier to reason about for security review.

Third, deployability. The frontend redeploys when I tweak Tailwind; the backend doesn't need to know. The backend redeploys when I change a route; the frontend keeps running. They scale independently. They restart independently. Same code split as production teams I respect.

Why MongoDB

The PRD recommends Mongo. I'd have picked Postgres for greenfield, but Mongo is a defensible choice for this domain — projects, tasks, attachments are a tree of mostly-self-contained documents, not a deeply normalized relational graph. The data access patterns lean read-heavy with frequent denormalization (tasks carry attachment summaries to avoid joins). Aggregation pipelines handle the activity feed cleanly. The driver and TypeScript story via Mongoose is mature.

The interesting downside is multi-document transactions: Mongo supports them via replica sets, but the dev story is fiddly. SYNC sidesteps the issue by keeping mutations small enough to live in one document, with two intentional exceptions (deleting a project cascades to tasks/files/activity/invitations; uploading a file appends to a task) where I accept eventual consistency on the audit log and best-effort cleanup on Cloudinary.

Why an object store separate from the API

Bytes don't belong in HTTP responses if you can avoid it. Cloudinary (or MinIO, in local Docker) serves the file directly via a signed URL; the backend just redirects with a 302. That keeps file traffic off the Render dyno (which has bandwidth limits on the free tier and CPU limits everywhere), and it lets the browser take advantage of CDN edge caches for image attachments. The API stays a thin orchestrator.

04Why each package, in detail

This section is exhaustive. For every package in either package.json, I explain what it does, what I considered instead, and why this one survived. If you're skimming, skip to the next section and come back when you want to argue with a choice.

Language & runtime

TypeScript (5.7 / 5.9)

Both services are TypeScript end-to-end, in strict mode. The brief doesn't require this — JavaScript would have been acceptable — but the cost of typing is negligible at this scale and the payoff is enormous: socket payloads, API request shapes, Zod-inferred validator types, and Mongoose document interfaces all flow through one type system. When I changed the Task model to add a version field for optimistic concurrency, the compiler walked me to every controller, service, and frontend store that needed updating. That's not a "nice to have" on a 48-hour timeline; it's how I stayed sane.

I considered using strict off for the frontend to move faster on shadcn boilerplate. I left it on. The shadcn primitives have well-typed Radix peers, and the cost was a half-dozen React.forwardRef generics — annoying for a minute, then never again.

Node 18+ (runtime)

The backend declares "engines": { "node": ">=18" }. Node 18 gives me native fetch, node:crypto.randomUUID, the Web Streams API, and a stable node:test runner (which I didn't end up using — see Jest below). Bumping to 20 wasn't justified; nothing I use needs it.

tsx (dev runner)

For npm run dev, I use tsx watch. It's a thin wrapper around esbuild + Node, which means transpile-only execution (no type checking on the hot path) and sub-second restarts on file change. The alternative is ts-node, which is slower and more configuration-heavy, or nodemon + a build step, which is two tools where one would do.

Type checking still happens — separately, via npm run typecheck (and the editor's language server). Splitting "run fast" from "type-check fully" is the modern Node TS pattern.

Backend libraries

Express (4.21)

The PRD names Express. I'd have considered Fastify for a greenfield project — it's faster and has a more modern middleware design — but Express has the largest middleware ecosystem in the Node world, and "the largest middleware ecosystem" matters when you're integrating helmet, rate-limit, express-validator, multer, swagger-ui-express, and cookie-parser inside a 48-hour window without surprises. Express's familiarity is also a signal: any reviewer can pick up the code immediately.

What I gave up by not using Fastify: a built-in schema validator (I use Zod instead, which is more flexible anyway), 20-30% raw throughput (irrelevant at the scale this app will see), and a cleaner async error model (mitigated by a asyncHandler wrapper that's twelve lines).

What I gave up by not using Nest: opinionated structure, dependency injection, decorators, modules. Nest is wonderful for large teams, but for two days and one developer it's a tax — class-based controllers and DI containers don't add anything when there are eight routes and no need for runtime substitution.

Mongoose (8.8)

An ODM, not just a driver. Schemas, instance methods, query builders, populate, virtuals, validation, middleware. I chose Mongoose over the raw mongodb driver because the schema-as-typed-document is exactly the shape I want — typed access for the service layer, server-side validation as defense-in-depth, and indexes co-located with the model so you don't drift between code and database state.

Alternatives I considered: Typegoose (Mongoose + decorators) is fine but adds runtime decoration overhead and isn't clearly better than hand-typed interfaces; the new Prisma + Mongo integration is interesting but the Mongo support is preview-quality and the aggregation pipeline story is weaker; raw mongodb driver is faster per call but requires hand-rolling everything from index management to populate-style joins.

Socket.io (4.8)

The PRD names Socket.io. Outside that constraint I'd have considered raw ws, which is faster and has no protocol overhead. Socket.io's killer features for this app are room-based broadcasts (built-in, with adapter support for Redis when you scale out), client-side automatic reconnection with exponential backoff, transparent fallback from WebSocket to long-polling for hostile networks, and acknowledgement callbacks for request-response over sockets.

The cost is a slightly larger payload (Socket.io's framing protocol adds a few bytes per message) and a heavier client bundle. Both are fine here. The room model alone saved me hours: I don't have to write a "fan out to subscribers" loop; I just join the project room on connect and emit to it from controllers.

bcryptjs (2.4) — and the bcrypt detour

Originally I used the native bcrypt package with cost 12. On Windows + pnpm during local dev, the native .node binding sometimes fails to build or unpack — and that's exactly what happened to my user (the famous "Cannot find module .../bcrypt_lib.node" error). I swapped to bcryptjs, a pure-JavaScript implementation with the same API. Slower for any individual hash (maybe 2-3× on cold benchmarks), but irrelevant: we hash exactly once per registration and once per password reset, and we compare exactly once per login. Throughput isn't the bottleneck; reliability is.

I also looked at argon2. It's the modern recommendation from OWASP, but the native build story is even fiddlier than bcrypt's, and the PRD specifically says "Password hashing with bcrypt (12+ salt rounds recommended)." Argon2 is a future migration, not today's choice.

jsonwebtoken (9.0)

Battle-tested JWT implementation. I considered jose, which is modern and supports the full JOSE suite (JWE, JWS, JWK) and is what I'd reach for in a system that needs encrypted tokens. SYNC uses HS256 signed tokens for access, refresh, reset, and invite — jsonwebtoken's simpler API matches that surface exactly.

cookie-parser (1.4)

Express middleware that parses the Cookie header into req.cookies. I need this because the refresh token lives in an HttpOnly cookie scoped to /api/auth, and the refresh endpoint reads it from there. Could have hand-parsed; this is twelve lines saved.

cors (2.8)

Express CORS middleware. I configured it with an allowlist callback so a misconfigured CORS_ORIGINS in production blocks requests loudly rather than silently allowing them. I also set credentials: true because the refresh cookie travels cross-origin from Vercel to Render in production, and that requires Access-Control-Allow-Credentials: true plus a non-wildcard origin.

helmet (8.0)

Sets sensible security headers — X-DNS-Prefetch-Control, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Strict-Transport-Security in production, Content-Security-Policy if you opt in. I left CSP off because configuring it correctly for Swagger UI and the Cloudinary image domains is more work than it's worth for the demo surface, and CSP is best added later with policy reports flowing into a monitoring system. I did override crossOriginResourcePolicy to cross-origin because the docs page is served from the same backend and may be loaded from a different origin during demos.

express-rate-limit (7.4)

In-memory sliding-window rate limiter with header support. I use two instances — a strict one on auth endpoints (10 per 15 minutes by default) and a permissive one globally (300 per 15 minutes). For a single-instance Render dyno, in-memory state is fine. For horizontal scaling you'd swap to rate-limit-redis. The skip: () => env.isTest escape hatch keeps tests from being throttled.

multer (1.4)

Multipart form parsing for file uploads. I configured it with memoryStorage so files never touch local disk on the Render dyno — they pass through a Node buffer and stream straight to Cloudinary or MinIO. The 5MB cap is enforced both here (limits.fileSize) and in my own assertAllowed guard (defense in depth — if multer config drifts, the guard still catches oversize).

multer-storage-cloudinary (4.0)

I listed this in the dependencies but I don't actually use it — I call Cloudinary's upload_stream directly from the storage adapter. Reason: the adapter pattern requires control over the resource type (image vs raw) and folder naming, and the storage adapter wrapper is cleaner with direct SDK calls. The package is harmless to leave in for the demo; in a cleanup pass I'd remove it.

cloudinary (2.10)

The official SDK. I lazily initialize it (the first call to getCloudinary() reads env and configures) so missing credentials in dev don't crash startup — we just log a warning and uploads fail at the actual upload time with a useful error. Same pattern for the S3 client.

@aws-sdk/client-s3 + s3-request-presigner (3.694)

For MinIO and any S3-compatible store. The v3 SDK is modular — I import only the two commands I use (PutObjectCommand, DeleteObjectCommand) and the getSignedUrl helper. The bundle is reasonable. I force path-style addressing because MinIO defaults that way, and AWS S3 supports it too. The presigner generates 5-minute download URLs with Content-Disposition: attachment for non-images so users get a download dialog instead of a maybe-renders-as-HTML preview.

resend (4.0)

HTTP-based email API. Simple SDK, no SMTP setup, free tier covers dev usage. I picked it over SendGrid because the API is cleaner and the free tier doesn't require billing setup, and over AWS SES because SES requires sandbox approval before you can email arbitrary recipients.

nodemailer (6.9)

For SMTP delivery in local Docker mode (talking to Mailpit). The classic Node mail library; works with every SMTP server in existence. I instantiate one transporter on first call, then reuse it.

zod (3.23)

Runtime schema validation library. Every request body, query, and param gets validated by a Zod schema before it reaches a controller. Zod's types are inferred (no separate TS types to maintain), and the same schemas drive the OpenAPI doc generation (via zod-to-openapi). That's the single biggest DRY win in the project — one declaration, two consumers.

I considered Joi (more verbose, no type inference), Yup (older, weaker inference), and class-validator + class-transformer (Nest-ecosystem heavy). Zod is the modern Node TS choice for good reason.

@asteasolutions/zod-to-openapi (7.3)

Lifts Zod schemas into an OpenAPI 3.1 document. I register paths against the same Zod validators the runtime uses, then generate the spec lazily on first request to /docs.json. Single source of truth for "what this endpoint accepts."

swagger-ui-express (5.0)

Serves Swagger UI from an Express middleware. I mounted it at /docs with persistAuthorization: true so the Bearer token survives page reloads — useful when you're click-testing flows during a demo.

pino (9.5) and pino-pretty (13.0)

Structured JSON logger, with a pretty-print transport for development. I picked pino over winston for three reasons: speed (pino is the fastest logger in Node by a wide margin), structured-by-default output (winston needs configuration to emit JSON), and the transport architecture for shipping to log aggregators in production.

node-cache (5.1)

In-process cache with TTL and key prefix iteration. My cache wrapper is forty lines of TypeScript that hides node-cache behind a Cache interface, so swapping to Redis later means writing one file. I considered lru-cache (no TTL out of the box) and a hand-rolled Map (no eviction). Both inferior for this surface.

ioredis (5.4)

Installed for forward compatibility — the cache contract is shaped to accept a Redis-backed implementation. I don't actually wire it up because the deployment target is single-instance and in-memory is sufficient. Listed in dependencies because the moment you scale to two backend instances, you need it and you don't want to be debugging a "cache miss on every request" symptom while figuring out which Redis library you forgot to add.

uuid (11.0)

I use it once, for the refresh token jti. node:crypto.randomUUID() would have worked too — I imported uuid for habit. In a cleanup pass I'd remove it and use the built-in.

xss (1.0)

Filter for HTML that may contain user-supplied content. I imported it for defense-in-depth on task descriptions, then realized I never render task descriptions as HTML on the frontend — they're plain text in a textarea and rendered with {description}, which React already escapes. The dependency is unused at the moment; harmless and ready for the day someone adds rich-text descriptions.

dotenv (16.4)

Reads .env into process.env at boot. I call it as the very first import in the env config so Zod's safeParse sees a populated environment. Standard.

Backend dev dependencies

jest (29.7) + ts-jest (29.2)

The PRD names Jest. Alternatives are Vitest (faster, ESM-native, would have been my greenfield pick) and Node's built-in test runner (lighter but less ecosystem). Jest's ecosystem won out — mongodb-memory-server has first-class Jest support, and supertest pairs naturally.

supertest (7.0)

HTTP request integration testing. Wraps an Express app instance and runs requests against it without binding a port. Every backend integration test in tests/*.test.ts uses it. It's the standard for Express testing for very good reason.

mongodb-memory-server (10.1)

Downloads a real Mongo binary and runs it in-process, in-memory, ephemeral. Each test run gets a fresh database; after each test, every collection is cleared. The alternative is a Docker-based test database, which is slower and requires Docker to be running on CI.

@types/* (various)

Type declarations for libraries that don't ship their own. Each is the obvious "@types/whatever the library is called" import.

Frontend libraries

Next.js (16.0)

App Router. The PRD asks for React Router, but Next.js gives me layouts, route groups, server components for the marketing page, code splitting at the route boundary for free, and a build pipeline that's just next build. The cost is a steeper learning curve and some patterns that change between versions (params became a Promise in Next 15+, which bit us during the upgrade — see the JWT section for the fix). Worth it.

Why I upgraded from 14 to 16 mid-build: the user's environment had a compatibility issue with how I wrote the project page (I had used the Next 15+ Promise-params pattern). Rather than rewrite to the legacy pattern, I bumped Next, which is now on 16 stable. The upgrade was painless because the code already targeted the new pattern.

React (19.0) and react-dom (19.0)

Required by Next 16. React 19 brings the ref-as-a-prop change (forwardRef is still supported but deprecated for new code), the use() hook for unwrapping promises (which is what consumes the params Promise in our project page), and slimmer Suspense behavior. Most of it is invisible to application code; the shadcn primitives still work because Radix UI 1.x updated for React 19 compatibility.

Tailwind CSS (3.4)

Utility-first CSS. The brief suggests Tailwind explicitly. Could have used Chakra, MUI, vanilla CSS modules — Tailwind wins for speed on a small team because there's no naming overhead, the design tokens flow from one config file, and the dark-mode story is one line. I'm on Tailwind 3 not 4 because shadcn components are still primarily 3-targeted as of writing; the migration to 4 is a follow-up.

shadcn/ui (inlined)

I didn't run the shadcn CLI — I hand-wrote each primitive into src/components/ui/ following the published patterns. That keeps me in control of versioning, lets me delete unused features without fighting a code generator, and means I can audit exactly what's in the bundle. The components I included: button, input, label, textarea, card, badge, avatar, dialog, select, skeleton, dropdown-menu. Everything else (popover, tabs, toast, separator, tooltip) is installed for completeness via Radix but not used in the current pages.

@radix-ui/react-* (various)

Accessible, unstyled primitives behind the shadcn components. Each Radix package is a peer of the matching shadcn component. They handle focus management, keyboard navigation, screen reader announcements, and portaling — the things that are easy to get wrong if you build your own dialog or dropdown.

class-variance-authority (0.7)

The variant API for Tailwind components. buttonVariants({ variant: 'destructive', size: 'sm' }) returns a className string. Cleaner than chaining cn() calls with conditionals.

clsx (2.1) and tailwind-merge (2.5)

The cn() helper combines them: clsx joins conditional classnames; tailwind-merge deduplicates conflicting Tailwind classes (so cn('p-2', 'p-4') resolves to p-4, not both). Used in every component.

lucide-react (0.469)

Tree-shakeable SVG icon library. I considered Heroicons (smaller, but fewer icons) and Tabler (more icons, slightly larger). Lucide is the default that shadcn uses and is well-maintained.

zustand (5.0)

State management. Three stores — auth, tasks, notifications. Zustand wins over Redux Toolkit for me on a small app because the boilerplate is minimal, selectors compose naturally, and there's no provider tree. The trade-off is no built-in time-travel debugging or middleware pipeline; if I were building Figma, I'd reach for Redux. For SYNC, Zustand is right-sized.

zod (3.23) on the frontend

Same Zod, with two roles: schemas for react-hook-form validation, and runtime parsing of API responses (defensive — server might lie, network might corrupt). The Zod types feed react-hook-form's resolver for type-safe form field names.

react-hook-form (7.54)

The PRD recommends it. I'd have picked it anyway — uncontrolled inputs by default (fewer rerenders), built-in resolver for Zod, intuitive register API. Used for every form in the app.

@hookform/resolvers (3.10)

The Zod resolver for react-hook-form. One import per form.

sonner (1.7)

Toast notifications. Tiny, accessible, no provider tree (just a Toaster component in the layout). I considered react-hot-toast (heavier) and the shadcn toast (more setup). Sonner wins on ergonomics.

socket.io-client (4.8)

Matches the server version. Automatic reconnection with exponential backoff is the killer feature — I configured infinite retry attempts so a disconnect during demos doesn't kill the session.

date-fns (4.1)

For the formatDistanceToNow calls in activity and notifications panels ("3 minutes ago"). Tree-shakeable; only the functions I import are in the bundle. I considered dayjs (smaller for plugin-heavy use) and Intl.RelativeTimeFormat (native, but needs date math). date-fns is right for this scale.

tailwindcss-animate (1.0)

Plugin that adds Radix-compatible animation utilities. Used by the dialog and dropdown-menu primitives for their open/close transitions.

autoprefixer, postcss

Standard Tailwind toolchain.

eslint, eslint-config-next

The default Next.js ESLint configuration. I didn't add custom rules — the project is small and the cost of bikeshedding a lint config exceeded the value at this scale.

Infrastructure choices

MongoDB Atlas (production) / mongo 7 in Docker (local)

Atlas free tier (M0) is enough for the demo. Backups, monitoring, IP allowlist UI. Atlas runs replica sets even on M0, which is necessary if you want to use transactions later. Local dev uses the official mongo:7 image with a volume for persistence.

Cloudinary (production) / MinIO in Docker (local)

Cloudinary's free tier covers the demo. MinIO is S3-compatible, runs in Docker, and is what I use for fully-local development. The storage adapter lets me switch between them with one env var.

Resend (production) / Mailpit in Docker (local)

Resend is the cleanest transactional email API I've used. Mailpit catches all SMTP traffic in a local web inbox at localhost:8025 so I can verify password-reset and invitation emails without sending real mail.

Render (backend)

Render's free tier gives me a long-running Node process — necessary for Socket.io. I shipped a render.yaml blueprint that provisions the service with all the env vars pre-declared, including generateValue: true for the JWT secrets so the first deploy gets fresh secrets without any manual work.

Vercel (frontend)

Default for Next.js. The Vercel CLI plus their GitHub integration means a push-to-deploy story with zero config.

pnpm

I used pnpm rather than npm for both projects. Reasons: faster installs, lower disk usage via the global content-addressable store, stricter peer-dep resolution. Trade-off: bcrypt's native binding was tricky to install on Windows + pnpm (the binary didn't unpack), which is why we swapped to bcryptjs. That was the only friction.

The folder structure follows a strict layering rule. Routes call controllers. Controllers call services. Services call models. Nothing calls upwards. Sockets and middleware are cross-cutting and may be called from either controllers or services (sockets from controllers when broadcasting after a mutation; middleware everywhere).

Why these particular boundaries

I treat the controller as a translation layer between HTTP and the domain. A controller decodes req, calls a service, then encodes the response — that's it. Business rules live in services, which means a service can be invoked from a controller, a socket handler, a CLI tool, or a future GraphQL resolver without copying logic. Models are passive data — schemas with indexes and methods, but no orchestration. This separation is what lets the tests run integration scenarios against the full HTTP surface and still feel readable.

5.2 Config layer

config/env.ts

This is the most important file in the backend. It's run at module load, parses process.env against a Zod schema, and exports a typed env object. If a required variable is missing or has the wrong shape, the process refuses to start with a useful error.

The schema covers everything: server settings, Mongo URI, four JWT secrets, four TTLs, cookie config, Cloudinary, S3, Resend, SMTP, rate limits, max file size. Every value is typed. Every coercion is explicit (z.coerce.number for numeric env vars, custom transforms for boolean strings).

I export a flag isProd and isTest derived from NODE_ENV, and the comma-separated CORS origins are split into an array at parse time so consumers don't have to keep re-splitting the string.

Why crash on bad env

A misconfigured deploy that runs anyway is a worse outcome than one that fails to start. If JWT_ACCESS_SECRET is missing in production, every login succeeds with a sign-of-empty-string token, and you'll find out about it three weeks later via a security report. Fail fast.

config/logger.ts

One pino instance. In development I wire up pino-pretty for colored output and friendly timestamps; in production I emit raw JSON so a log aggregator (Logtail, Datadog, etc.) can parse it. The base context attaches service: 'sync-backend' to every log line, so even if I add a frontend service later that ships logs to the same place, they're filterable.

config/db.ts

Two exports: connectDB and disconnectDB. The connect function sets strictQuery: true globally (Mongoose 7+ recommendation — refuses queries that reference fields not in the schema), waits up to 10 seconds for connection, and disables autoIndex in production.

Why disable autoIndex in production

By default Mongoose builds indexes on connection. That's fine in dev — it's how I make sure my .index({ ... }) calls actually create indexes. In production it's a foot-gun: a deploy that adds a new index can trigger a long-running index build on a large collection, holding the deploy hostage. The recommended pattern is to manage indexes via migrations or out-of-band scripts. For SYNC I leave autoIndex on in dev so I see the index work locally, and off in prod where the indexes are already there from the first deploy.

config/cloudinary.ts

Lazy initialization. The first call to getCloudinary() configures the SDK with the env credentials and caches the client. If credentials are missing, we log a warning but still return the client — uploads will fail with a useful Cloudinary error at the time of upload, not at boot.

This matters in local Docker dev: STORAGE_PROVIDER=s3 means we never call getCloudinary() at all, so missing Cloudinary credentials are silently OK.

5.3 Utilities

utils/errors.ts

A small class hierarchy: AppError as the base, with BadRequestError, UnauthorizedError, ForbiddenError, NotFoundError, ConflictError, ValidationError, and RateLimitError as subclasses. Each carries an HTTP status, a machine-readable code (BAD_REQUEST, NOT_FOUND, etc.), an optional details payload, and the isOperational flag that I use to distinguish "expected" errors from programmer bugs.

The advantage of an error hierarchy: services and middleware throw new ConflictError('User exists') without caring about response shape, and the central error handler maps each subclass to the right status code and JSON. The advantage of isOperational: in a future iteration with Sentry or similar, I can filter out the expected 4xx noise.

utils/asyncHandler.ts

A twelve-line wrapper that catches promise rejections in async route handlers and passes them to Express's next. Without it, every controller would need a try/catch with next(err). With it, the controller body is just business code.

export const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

Express 5 will handle async errors natively, but Express 4 (the version we're on) doesn't.

utils/apiResponse.ts

Three helpers — ok, created, noContent — that send responses in a consistent { success, data, meta? } envelope. Error responses come from the error handler with { success: false, error: { code, message, details? } }. The envelope is documented in docs/API.md and reflected in the OpenAPI spec.

I considered going envelope-less (just return the data, use HTTP status for success/failure). The envelope wins because pagination meta has to live somewhere, and inventing one-off response shapes for "list with pagination" vs "single resource" creates inconsistency. With the envelope, every endpoint looks the same.

utils/jwt.ts

One sign/verify pair, four token kinds — access, refresh, reset, invite. Each kind has its own secret and TTL (read from env at module load). The function signatures are typed with discriminated unions on the type field, so a controller that mistakes a reset token for an access token gets caught at compile time.

Why four separate secrets: blast-radius isolation. If the access secret is somehow leaked, reset and invite tokens are still trustworthy. In a future iteration these would rotate independently via a key-management service.

utils/ms.ts

A tiny duration parser — accepts 15m, 30d, 3600s, etc. I wrote it inline rather than importing the ms npm package because it's eleven lines of code and dropping a tiny dep keeps the surface auditable. The auth service uses it to compute refresh-cookie expires times from the configured TTL string.

5.4 Middleware

middleware/errorHandler.ts

Central error handler, registered last in the Express pipeline. It maps known errors to the right status/code/message: AppError subclasses pass through, ZodError becomes 422, mongoose.Error.ValidationError becomes 422, mongoose.Error.CastError becomes 400, the 11000 duplicate-key Mongo error becomes 409, JWT errors become 401, and anything else becomes 500.

The output shape is the inverse-envelope. Stack traces appear in development; in production they're omitted to avoid leaking internals. Server errors (5xx) get logged at error level; client errors (4xx) at warn.

There's also a notFound handler that responds to any path that didn't match a route with a 404 and a useful message. Mounted before the error handler.

middleware/validate.ts

A factory that takes a Zod schema and a source (body, query, or params) and returns a middleware. The middleware calls schema.safeParse, and on failure it forwards the ZodError to next (which the error handler converts to a 422). On success it overwrites req[source] with the parsed/cleaned value, so downstream handlers see sanitized data.

This is the only validation layer. Mongoose schemas also validate, but as defense-in-depth — the express layer is the source of truth for request shape, and errors that escape Zod are bugs rather than user input.

middleware/rateLimit.ts

Two instances of express-rate-limit: authLimiter with a tight window (10 per 15 minutes by default) applied to the auth routes, and globalLimiter with a permissive window (300 per 15 minutes) applied to the whole API. Both skip in test mode.

The auth limiter is per-IP (default). For a production rollout I'd add per-account limits — five failed logins on an account locks it for an hour — but per-IP is the right baseline and the brief mentions brute force prevention which this addresses.

middleware/auth.ts

Extracts the Bearer token from Authorization, verifies it as an access token, and attaches req.user = { id, email }. On any failure it throws UnauthorizedError. Twelve lines.

The middleware doesn't load the full user from the database. req.user carries the minimum needed for authorization decisions (the id, which is the join key for every project membership lookup). Loading the user from the DB on every request would add a round-trip per call; the access token's freshness window (15 minutes) is an acceptable staleness budget for things like display name and email.

middleware/rbac.ts

The single most important middleware in the project. requireProjectRole(minRole) reads the project id from the URL parameter, loads the project's owner and members from Mongo, resolves the caller's role (owner is implicitly admin), and checks it meets the minimum required role. On success it attaches req.projectRole and req.projectId so downstream controllers can branch on role if needed.

The role rank table is a constant: { viewer: 0, member: 1, admin: 2 }. Higher beats lower. Every project-scoped route mounts this middleware before its controller; the controller never needs to second-guess whether the caller is allowed in.

Why the middleware loads the project

The alternative is to check membership inside each controller — but then every controller does the same query, and any controller that forgets is a privilege-escalation bug. By centralizing in middleware, the check happens exactly once and can't be skipped. The cost is one Mongo round-trip per project-scoped request, mitigated by a future cache layer (the cache contract is already there; I just don't cache memberships yet because they change rarely and the query is fast with the index on members.user).

middleware/upload.ts

Multer with memory storage and a 5MB limit. Five files max per request, although every route only accepts one. The buffer flows from multer into the storage adapter; we never touch local disk.

5.5 Models

models/User.ts

The user document. Email is unique, lowercased, trimmed, and indexed. Password is stored as a bcrypt hash with select: false so it's never returned by default — to fetch it for login, the auth service explicitly calls .select('+passwordHash'). The instance method comparePassword wraps bcrypt's compare; the static hashPassword wraps bcrypt's hash with cost 12. A toJSON transform strips the password hash defensively in case it ever leaks into a query result.

I considered storing additional profile fields (timezone, locale, preferences). Cut for scope — not in the brief, and trivial to add later.

models/RefreshToken.ts

One row per active session. Stores the user reference, the JWT id (jti), the bcrypt hash of the token (so a database leak doesn't give an attacker valid refresh tokens), the user agent and IP for audit, the expiry, and revoked/replaced metadata for rotation chains. A TTL index on expiresAt means Mongo purges expired rows automatically — no cron needed.

The model's existence is itself a security decision: server-side state lets us revoke individual sessions and detect token theft. A purely stateless JWT-only setup is faster but can't do either.

models/Project.ts

Name, description, status, owner, and an embedded array of { user, role, joinedAt } member records. Embedded rather than a separate collection because the read pattern is "give me the project with its members" 99% of the time, and embedding avoids the join (Mongo doesn't natively do joins anyway, but the populate-style lookup has the same overhead).

Two indexes: one on owner (for "projects I own") and a compound one on members.user + status + updatedAt (for "active projects I'm a member of, sorted by recent activity"). The compound is what the dashboard query uses.

models/Task.ts

The most complex model. Reference to project, title, description, status enum, priority enum, array of assignee references, optional due date, an embedded array of attachment summaries, references to creator and updater, a version integer for optimistic concurrency, and timestamps.

Indexes: project alone, project + status + priority (kanban view), project + updatedAt (recent activity sort), project + dueDate (calendar view), assignees (filter by assignee), and a text index on title + description for search.

The embedded attachment summaries duplicate information from the File collection. Reason: the kanban renders task cards with attachment counts and sometimes a thumbnail; doing that without a join means each task carries everything it needs to render. The trade-off is consistency — if I update a file's URL (rare), I'd need to update every task that embeds it. For SYNC the only mutations to attachments are add and remove (which the file service handles atomically), so the embed is safe.

models/Activity.ts

Append-only log of significant project events. Each row carries project, actor, action enum, optional target, optional metadata. Indexed by project + createdAt (descending) for the activity feed query. Cursor-paginated by _id (which is monotonically increasing).

models/Invitation.ts

One row per pending invitation. Project, email, role, jti (the JWT id baked into the invite token), inviter, expiry, optional acceptance metadata. TTL index on expiresAt auto-purges expired invitations. The model exists for two reasons: to support invitation revocation (a future feature — set revokedAt), and to defend against replay (re-accepting an already-accepted invitation should fail with a clear error rather than silently mutate state).

models/Notification.ts

User, kind, optional project/task/actor references, message, read flag. Indexed by user + read + createdAt so the bell icon's "unread count" query and the "recent notifications" query both hit the same index efficiently.

models/File.ts

Per-upload record. Project, optional task, filename, mime type, size, URL, public id, resource type, uploader. The public id is the provider-internal identifier — Cloudinary returns one shape, MinIO uses the S3 object key. Either way it's what we use to delete the file later.

5.6 Services

services/authService.ts

The auth service is where most of the security thinking lives. It exports five functions: registerUser, loginUser, rotateRefreshToken, revokeRefreshToken, requestPasswordReset, resetPassword, plus the internal issueTokens.

registerUser checks email uniqueness, hashes the password, creates the user, then issues a token pair. The token pair includes an access JWT signed for 15 minutes and a refresh JWT signed for 30 days; the refresh JWT's bcrypt hash is stored in the database along with its jti, the user agent, IP, and expiry.

loginUser looks up the user with +passwordHash, compares the password (throws on mismatch with a generic error to avoid email enumeration), and issues tokens. The "wrong email" and "wrong password" cases return the same 401 message.

rotateRefreshToken is the most security-sensitive function in the codebase. The flow is: verify the JWT signature (throw 401 if invalid), look up the stored row by jti (throw 401 if missing), detect token reuse (if the stored row is already revoked, revoke every refresh token belonging to the user and throw — this is the "stolen token detected" branch), bcrypt-compare the presented raw token against the stored hash (throw 401 on mismatch), load the user (throw 401 if deleted), issue a new pair, mark the old row revoked with replacedByJti pointing at the new jti.

revokeRefreshToken is idempotent — used by logout. It tries to verify, then marks the row revoked. Verification failures are swallowed (a malformed cookie shouldn't cause logout to fail noisily).

requestPasswordReset looks up the user but always returns success — the message says "if an account exists, we sent a link." This prevents email enumeration. If the user exists, a signed reset token is generated (with the dedicated JWT_RESET_SECRET) and emailed via the email service.

resetPassword verifies the reset token, hashes the new password, updates the user, and revokes every refresh token for the user — so changing your password kicks out every existing session, which is the right behavior.

services/projectService.ts

Owns the project lifecycle and member management. Functions: listProjectsForUser (paginated with cache), getProjectById, createProject (caller becomes admin/owner), updateProject, deleteProject (owner-only, cascades to tasks/files/activity/invitations), inviteMember (creates invitation row + emails token), acceptInvitation (verifies token, adds membership), changeMemberRole, removeMember.

Two patterns are worth calling out. First, every mutation calls logActivity for the audit trail. Second, every mutation calls invalidateUserProjectCache so the cached "my projects" list refreshes on the next read.

The acceptInvitation function checks three things in order: the JWT signature, that the invitation row exists and isn't revoked or already accepted, and that the accepting user's email matches the invitee's email. That last check is what prevents an attacker who steals the invitation link from joining as their own account — the link is bound to the original recipient's email.

services/taskService.ts

Task CRUD with optimistic concurrency. The key function is updateTask, which accepts an expectedVersion. The Mongoose call is findOneAndUpdate({ _id, project, version: expectedVersion }, { $set: ..., $inc: { version: 1 } }). If expectedVersion doesn't match, the filter matches zero documents and Mongoose returns null. We then check whether the task exists at all (to distinguish "version conflict" from "not found") and throw the appropriate error.

List queries support filtering, sorting, search, and pagination. The cache is keyed on a tuple of (project, status, priority, assignee, search, sort, page, limit) so the same query hits the same cache slot. Cache TTL is 15 seconds, which is short enough that staleness is rarely user-visible and long enough to absorb a burst of identical queries during a busy project view.

Bulk update uses updateMany with $inc: { version: 1 } so versions stay coherent. Bulk delete uses deleteMany. Both invalidate the cache and log activity.

The service also handles assignee notifications: when a task is created or its assignees change, every assignee (except the actor) gets a Notification row written. The notifications are surfaced via the bell icon and via Socket.io broadcasts (the controller emits a task:created event to the project room, which the client uses to update the kanban; in a future iteration I'd also emit a per-user notification event to the assignees' personal rooms).

services/activityService.ts

logActivity is a fire-and-forget create with a top-level try/catch that swallows errors. The reasoning: an activity log failure should never roll back a parent mutation. If the activity log is briefly unavailable, the user shouldn't see "failed to create task" — the task should still be created and we'll deal with the gap in the audit feed.

listActivity uses cursor pagination — the cursor is the last seen _id and the query is { project, _id: { $lt: cursor } }, sorted descending. This pattern beats offset-based pagination on append-heavy collections because new inserts don't shift the existing pages.

services/fileService.ts

File uploads, downloads, deletes — with allow-listed MIME types and a 5MB cap enforced by assertAllowed. The upload function calls the storage adapter (which routes to Cloudinary or S3 based on env), persists a File document, and if the upload is attached to a task, atomically pushes the attachment summary onto the task's array and bumps the task version.

Downloads are served via signedDownloadUrl, which returns a short-lived URL the controller redirects to with a 302. The Cloudinary path uses private_download_url with a 5-minute expiry; the S3 path uses getSignedUrl with the same window and a Content-Disposition: attachment header for non-images.

Delete checks that the caller is either the uploader or an admin, deletes from the storage provider (best-effort — failures are logged), and pulls the attachment off any task that references it.

services/storage/* (adapter pattern)

The storage layer is the cleanest abstraction in the codebase. Three files plus a barrel: types.ts defines the StorageProvider interface with upload, delete, and signedDownloadUrl; cloudinaryStorage.ts and s3Storage.ts implement it; index.ts picks one based on STORAGE_PROVIDER and caches the choice.

This is the file structure I'd reach for if I were building a service that needed to support both AWS S3 and a self-hosted MinIO from day one, or if I were anticipating a vendor switch. The cost of writing it was an hour; the benefit is that the local Docker dev story doesn't require Cloudinary credentials at all.

services/emailService.ts and services/email/*

Same shape, applied to email. The top-level emailService.ts owns the template builders (passwordResetEmail, invitationEmail) — those are pure functions returning { subject, html }. The actual delivery happens via the adapter in services/email/index.ts, which picks between Resend, SMTP (nodemailer), and a console fallback that logs to stdout when no provider is configured.

The console fallback is what makes the dev experience pleasant when you haven't configured email yet — instead of "Failed to send" toasts, you see the reset link in your terminal output and can click it manually.

5.7 Controllers

controllers/authController.ts

Thin wrappers around the auth service. The interesting work is around cookies — the setRefreshCookie helper sets HttpOnly, Secure (in production), SameSite=None (cross-origin) or Lax (dev), path /api/auth, and expiry matching the refresh token's expiry. clearRefreshCookie does the inverse for logout.

The refresh handler reads the cookie (or the body, for flexibility), calls rotateRefreshToken, and sets the new cookie + returns the new access token. The logout handler revokes the token and clears the cookie. Both are idempotent.

controllers/projectController.ts

List, detail, create, update, delete, invite, accept-invite, set-member-role, remove-member, activity. Each is a five-to-fifteen-line function that decodes req, calls the service, and either calls emitToProject with the right event or returns the response. All mutating endpoints emit a socket event so connected clients update their UI without polling.

controllers/taskController.ts

List, detail, create, update, delete, bulk-update, bulk-delete. The update handler passes expectedVersion through from the body to the service, so the version conflict error bubbles up naturally as a 409. The bulk endpoints accept arrays of ids and a patch object.

controllers/fileController.ts

Upload, list, download (302 redirect), delete. The upload handler reads req.file from multer, optionally req.body.taskId for attachment, and hands off to the storage adapter via the file service.

routes/auth.ts

r.post('/login',            authLimiter, validate(loginSchema),           ctrl.login);
r.post('/refresh',                                                        ctrl.refresh);
r.post('/logout',                                                         ctrl.logout);

Each route shows its middleware chain inline. Auth limiter on every endpoint that takes user input; Zod validation everywhere a body is consumed; requireAuth on the one protected route. The arrangement makes the security posture inspectable at a glance.

routes/projects.ts

List and create at root, with requireProjectRole on every :projectId-scoped route. The accept-invite endpoint is outside the project scope (it's how you become a member, so you can't be required to be one first).

routes/tasks.ts and routes/files.ts

Both mounted under /api/projects/:projectId/... with mergeParams: true so the inner router sees the projectId from the parent. The same RBAC middleware applies. Bulk endpoints come before :taskId routes so Express doesn't accidentally interpret "bulk-update" as a task id.

routes/notifications.ts

List (with unread count in meta) and mark-read (with optional ids — empty marks all read).

5.9 Sockets

sockets/index.ts

The Socket.io server is initialized with CORS matching the REST API, both transports enabled (WebSocket preferred, long-polling fallback), and a typed SocketData shape. The handshake middleware verifies the JWT exactly the same way the HTTP middleware does — we share the verification logic to avoid two paths that could drift.

On connect, the socket joins a personal room named user:<id> (for notifications targeted at one user across all their devices). The client then emits project:join with a project id; the server re-checks membership against the project and joins the socket to the project:<id> room. The re-check is crucial — even though the user can't access the project via REST without RBAC passing, the socket is a separate channel and we don't trust the client to tell us what rooms they can join.

Presence: when a user opens a task dialog, the client emits task:editing. The server records the editing state on the socket and broadcasts to everyone else in the room. On disconnect, if the user was editing a task, the server emits task:stop_editing on their behalf so other clients clear their "X editing" badge.

sockets/emit.ts

Two helpers — emitToProject(projectId, event, payload) and emitToUser(userId, event, payload). The Socket.io server instance is stored at module level after init, so controllers can call these without dependency injection. Yes, this is a hidden singleton. Yes, for a system with one io server it's the cleanest pattern.

5.10 Cache

cache/memoryCache.ts

A thin wrapper around node-cache exposing a Cache interface with get, set, del, invalidatePrefix, and flush. Default TTL is 30 seconds.

The invalidatePrefix function iterates all keys and deletes those matching the prefix. It's O(n) over the keyspace, which is fine at this scale. Used for "invalidate every cached query for this user's projects" or "every cached task list for this project."

What I cache: project list per user (key includes filters + page), task list per project (same). What I deliberately don't cache: individual project detail, individual task detail, user profile, activity feed. Reason: those queries are already fast (indexed point lookups) and caching them creates invalidation complexity for marginal benefit.

5.11 Validators

validators/auth.ts

Zod schemas for register, login, forgot password, reset password. The register and reset schemas enforce password complexity (8+ chars, upper, lower, number) via three refine calls.

validators/project.ts

Schemas for project create, project update, invite, accept-invite, member role, member id param, activity query, project list query. Also the shared objectId regex for Mongo ids.

validators/task.ts

Schemas for task create, task update (with optional expectedVersion), task list query (with sort enum), bulk update, bulk delete.

All three files are the input contract for the API. Their TypeScript types (via z.infer) flow through services. They're also what powers the OpenAPI generation.

5.12 OpenAPI docs

docs/openapi.ts

Builds the OpenAPI 3.1 document by registering every path against the Zod validators. Each registered path declares its method, URL, tags, summary, security scheme reference, request shape (body/query/params), and response shapes.

The same Zod schemas used at runtime to validate inputs are converted to OpenAPI request schemas via @asteasolutions/zod-to-openapi. Response schemas are written separately as "DTOs" (UserRef, Project, Task, Activity, etc.) so the docs reflect what the API actually returns, not what the database stores.

The Bearer security scheme is declared once and referenced from every protected path.

docs/swagger.ts

Mounts swagger-ui-express at /docs with persistAuthorization: true. Also exposes the raw spec at /docs.json so anyone can import it into Postman, Insomnia, openapi-generator, etc.

tests/setup.ts

tests/auth.test.ts

Eight cases: register success, duplicate registration rejected, login with correct credentials, login rejected with wrong password, password is hashed (not stored plain), password complexity enforced, refresh works via cookie, /me protected with bearer.

tests/tasks.test.ts

6.2 Tailwind & theme

Standard shadcn-style configuration with CSS variables for colors. Light and dark token sets in globals.css. The tailwindcss-animate plugin adds the keyframes that Radix's data-state-driven animations expect.

The colors are tokenized — hsl(var(--primary)) rather than fixed values — so changing the theme means swapping CSS variables, not chasing usages. Dark mode is class-based (darkMode: ['class']) which leaves the toggle implementation up to the app; SYNC doesn't yet have a dark-mode switch in the UI (cut for scope) but the tokens are ready for it.

Three Tailwind directives (@tailwind base/components/utilities), the light/dark CSS variables in :root and .dark, and a base layer that applies the border color globally and sets background/foreground on body. Twenty lines of CSS total — all the rest is utility classes in components.

6.3 Shadcn primitives

Each primitive is a thin styled wrapper around a Radix primitive (where one exists) or a styled HTML element. The pattern across all of them: a cva variants definition for size/color modes, a forwardRef wrapper for ref passthrough, a cn(...) call combining base classes, variant classes, and any caller-provided className.

6.4 Stores (Zustand)

store/auth.ts

Three pieces of state: the access token (in-memory only — never in localStorage), the current user, and a hydrated flag indicating whether the auth provider has finished its initial refresh attempt. Setters for each plus a clear that resets token and user (used on logout).

Why the hydrated flag: the app layout uses it to decide whether to redirect to login. Without it, every page mount would briefly redirect to login while the silent refresh runs, then redirect back — a visible flash.

store/tasks.ts

The most interesting store. Tasks are normalized as byProject: Record<projectId, Record<taskId, Task>>. Presence is shaped similarly: presence: Record<projectId, Record<taskId, Set<userId>>>. Actions: setMany (replace tasks for a project on initial load or filter change), upsert (insert or merge a single task — used by socket handlers), remove, applyBulkPatch, bulkRemove, startEditing, stopEditing, clearProject.

The upsert function carries a version check — if the incoming task has a lower version than what we have, we discard it. That's the last-write-wins resolution that handles racy socket events where two updates arrive close together.

The Zustand-with-React-19 gotcha I hit: returning a fresh {} from a selector on every call trips React's "infinite loop" guard. The fix is a module-level EMPTY_TASK_MAP constant returned as the stable fallback. Worth knowing because it's a footgun for any new selector that has an "or empty" branch.

store/notifications.ts

Items array + unread count + actions setAll, prepend (for new socket-pushed notifications), markAllRead. Simple.

6.5 API client

lib/api.ts

The single most important file in the frontend. It's a typed fetch wrapper with three responsibilities: attach the access token from the auth store, send credentials (so the refresh cookie travels), and handle 401 by triggering a single-flight refresh and replaying the request.

The single-flight refresh is a module-level Promise<string | null> | null. If a request gets a 401 and no refresh is in flight, we start one; otherwise we await the existing one. This means a burst of N parallel 401s doesn't fire N refresh requests — exactly one runs and everyone shares the result.

On successful refresh we update the auth store with the new token and replay the original request with the new Authorization header. On failure we clear auth and let the calling code render an unauthenticated state.

Two exported variants: api<T> returns the data payload unwrapped (for the common case), and apiRaw<T> returns { data, meta } for endpoints with pagination.

For file uploads we use uploadWithProgress — an XHR-based helper, because fetch can't report upload progress. The progress callback drives the progress bar in the task dialog.

6.6 Socket client

lib/socket.ts

A module-level singleton. getSocket() returns the current connection, creating one if needed. If the access token rotates (after a refresh), the next getSocket() call disconnects the old socket and creates a new one — sockets are bound to a token, and a refresh invalidates the old binding.

Joined rooms are tracked in a module-level Set. On reconnect (Socket.io fires connect after a backoff retry succeeds), we emit project:join for every project the user was in. This means a brief network blip doesn't permanently drop the user out of their project rooms.

The client also subscribes to every server event: task:created, task:updated, task:deleted, the bulk variants, the presence events, the project events, and notification. Handlers dispatch to the appropriate Zustand action.

6.7 Auth provider

components/providers/auth-provider.tsx

Mounted in the root layout. On mount it fires a single POST /api/auth/refresh with skipAuth: true (we don't have an access token yet). If it succeeds, we get a fresh access token from the refresh cookie, then fetch /me to populate the user. Either way, we set hydrated: true so the rest of the app can render.

This is what makes "stay logged in across reloads" work. The refresh cookie is HttpOnly and persistent (30 days); on every cold boot the auth provider trades it for a new access token.

6.8 App layout

The route group (app) wraps every authenticated route. The layout reads hydrated and user from the auth store: if not hydrated yet, render a skeleton; if hydrated and no user, redirect to login; otherwise render the chrome (top bar with notifications bell and user dropdown) and the page below.

07JWT, in detail

If there's one section to read carefully, it's this one. JWT design is where most authentication footguns live. SYNC's setup is deliberately conservative.

Why two tokens

The pattern is access + refresh. The access token is short-lived (15 minutes) and stateless — every signed access JWT is valid until expiry, with no database check. The refresh token is long-lived (30 days), stored server-side as a hash, and required to obtain new access tokens.

This gives us two properties at once. Access checks are cheap (one JWT verify, no DB round-trip), which is important for high-throughput endpoints and for sockets where every event would otherwise hit the database. Revocation is possible (the refresh row can be marked revoked, denying further rotation), which is what makes "log out from all devices" a real feature rather than a wish.

Why short access TTL

The blast radius of a leaked access token is bounded by the TTL. 15 minutes is a sweet spot — long enough that a user isn't constantly noticing refresh round trips, short enough that a casually stolen token expires before it's useful. Some teams go to 5 minutes. I'd go shorter if the app were higher-stakes; for project management 15 is appropriate.

Why store refresh tokens server-side

Two reasons. First, revocation: a pure-stateless refresh would be valid for its full lifetime regardless of logout. Second, theft detection.

The theft detection mechanism is the most interesting bit. When a refresh succeeds, the old refresh row is marked revokedAt and the new row is created with replacedByJti pointing at the new jti. If a refresh comes in with a token whose row is already revoked, that's either (a) a stale token sent by accident, or (b) a stolen token being used by an attacker after the legitimate user already rotated.

We can't tell which it is from a single request, but we can take the safe action: revoke every refresh token for that user. The legitimate user gets logged out and has to log in again (annoying but bounded); the attacker can't continue rotating. This is RFC 6819's recommended pattern (and the basis for OAuth 2.0 refresh token rotation in section 5.2.1 of RFC 8252).

Why bcrypt-hashed refresh tokens

A database leak that included plaintext refresh tokens would let an attacker forge sessions until each token expires. With bcrypt hashes, even a full database leak doesn't yield usable tokens — the attacker would have to brute-force each hash, which is by design slow.

The cost is one bcrypt operation per refresh request. At cost 10 (lower than password hashing) this is about 50ms. Acceptable for a request that happens every 15 minutes.

Why HttpOnly cookies for refresh, in-memory for access

The refresh token is high-value (long-lived, can spawn access tokens). Putting it in localStorage exposes it to any XSS. Putting it in an HttpOnly cookie hides it from JavaScript entirely.

The access token is lower-value (short-lived, single-purpose) and needs to be readable by the JS that makes API calls. The compromise is to keep it in memory only — never in localStorage, never in cookies. On page reload it's gone, and we recover by trading the refresh cookie for a new access token during auth provider hydration. This is the "in-memory access + HttpOnly refresh cookie" pattern, which is the OWASP recommendation for SPAs.

The cookie attributes

The refresh cookie carries these attributes:

  • HttpOnly — invisible to JS.
  • Secure in production — only sent over HTTPS.
  • SameSite=None in production (when frontend and backend are different origins, which they are on Vercel + Render) — required for cross-origin cookie transport. Browsers require Secure for SameSite=None, hence the previous bullet.
  • SameSite=Lax in dev — cookies travel on top-level navigations and most cross-origin requests, which is convenient for local testing.
  • Path=/api/auth — the cookie is only sent on auth routes. This limits exposure: even if a non-auth route is somehow vulnerable to CSRF, the refresh cookie isn't in scope.
  • expires matching the refresh TTL.

Four secrets, four token kinds

Access, refresh, reset, invite each have their own signing secret. If the access secret were ever leaked, the attacker could forge access tokens but not reset or invite tokens. This is paranoia in the good sense — separating concerns reduces the impact of any single failure.

The secrets are generated fresh on each Render deploy via generateValue: true in render.yaml. Locally they're filled with high-entropy random values in .env.

Reset tokens revoke sessions

Resetting a password invalidates every refresh token belonging to the user. The reasoning: if the user is resetting their password, they likely suspect compromise, and the right action is to terminate every existing session. The next login will produce a new pair.

Invitation tokens are bound to email

Invitation JWTs carry the invitee's email in the payload. On accept, the server verifies that the accepting user's email matches the invited email. This prevents an attacker who steals an invitation link from joining as their own account — the link is bound to the original recipient.

08Caching, in detail

What's cached

Two endpoints. The "list my projects" query (per user, per filter set) and the "list tasks in this project" query (per project, per filter set). Both are paginated, both can be hot during dashboard browsing, both can be invalidated cleanly when the underlying data changes.

What's not cached

Individual project detail, individual task detail, user profile, activity feed, notifications. Reason: those are either small point lookups (cheap to recompute) or write-heavy (caching adds invalidation complexity for marginal benefit).

The cache key shape

For projects: projects:<userId>:<status>:<search>:<page>:<limit>. For tasks: tasks:<projectId>:<status>:<priority>:<assignee>:<search>:<sort>:<page>:<limit>. The components are concatenated in a fixed order; missing values are empty strings.

Two queries with the same filters resolve to the same key. Two queries that differ in any way produce different keys. The downside is keyspace explosion — every distinct filter combination is its own cache entry, which can chew through memory if users hit many distinct filter sets. Mitigated by the short TTL.

Invalidation by prefix

Every project mutation calls memoryCache.invalidatePrefix('projects:' + userId), which iterates the keyspace and deletes every key starting with that prefix. Every task mutation calls the analogous task-prefix invalidation.

This is the simplest correct invalidation. Smarter strategies (compute which cache keys are actually affected and only invalidate those) save memory but add complexity. Prefix invalidation is O(n) over keys, which is fine at the scale this app runs at.

TTLs

Project list: 30 seconds. Task list: 15 seconds. These are short enough that staleness is rarely user-visible and long enough to absorb bursts of identical requests during a busy session.

Why in-memory, not Redis

Two reasons. First, the deployment target is a single Render dyno. Multiple dynos would need a shared cache; one dyno doesn't. Second, the Cache interface is abstract enough that swapping to Redis is a one-file change — I haven't done it yet because there's no scaling pressure, but the path is short.

I installed ioredis as a dependency anyway, signaling intent and avoiding the situation where someone discovers the missing dependency mid-migration.

Cache coherence with sockets

A subtle point: when a task is updated via REST, the controller broadcasts task:updated to the project room. Clients connected to that room update their local store immediately. Other clients (not in the room, browsing the project list) might hit a cached "task list" query and see stale data for up to 15 seconds.

That's acceptable. The cache is a freshness optimization, not a correctness mechanism. The socket layer carries the freshness signal to anyone actively viewing the project.

09Real-time, in detail

The room model

Two kinds of rooms. The personal room user:<id> is joined on connect and carries notifications addressed to that user — useful when the user is connected from multiple tabs or devices. The project room project:<id> is joined explicitly via the project:join event after the server re-checks membership.

Broadcasts go to project rooms (for project-scoped events) or personal rooms (for direct-to-user notifications). The two are independent: a user can be in zero projects yet still receive personal notifications.

Authorization on join

The handshake authenticates the user (JWT). Joining a room authorizes them for it. The server re-runs the membership check on every project:join — we don't trust the client to send a project id they're allowed in.

This is the defense against the "user used to be a member, was removed, but their socket still in the room" scenario. The room subscription is established at join time, and the membership check at that moment is what gates access. There's a residual window: if a user is removed mid-session, they'll continue to receive events until they disconnect. A future improvement is to track membership changes and force-leave affected sockets.

Server is the single source of truth for writes

I made this a non-negotiable rule. Clients never write to the database via sockets. Every state change goes through an HTTP endpoint with full validation, RBAC, and audit logging. After the change lands, the controller broadcasts the new state to the project room.

The opposite pattern — clients emit "task:update" with a payload, the server applies it directly to the database — is faster (no HTTP round trip) but creates an entirely separate validation pipeline. I would have to duplicate every Zod check inside the socket handler, plus the RBAC check, plus the cache invalidation. The complexity isn't worth the latency saving.

Presence indicators

The "X is editing this task" badge is implemented with two events: task:editing when the client opens a task dialog, task:stop_editing when it closes. The server broadcasts each to the project room.

The cleanup-on-disconnect is the important bit. If a user closes their browser mid-edit, the server inspects the socket's stored editing state and emits task:stop_editing on their behalf. Without this, other clients would see a ghost presence badge until the next page reload.

Optimistic concurrency, end to end

Tasks carry a version integer. Every update increments it server-side via $inc. Client updates pass an expectedVersion; if it doesn't match the stored version, the conditional update matches zero documents and we return 409 with the code VERSION_CONFLICT.

The client handles 409 by refetching the task and surfacing a toast ("This task was modified — please reload"). The user sees the latest state and can re-apply their change.

For socket-driven updates, the client's store has the same version check in its upsert: if the incoming task has a lower version than what we have, we drop the update. This handles racy event delivery where an old update arrives after a newer one.

What this doesn't do

Operational transformation (Google Docs-style real-time collaborative editing of the description field) is not implemented. The current model is last-write-wins on the whole task. Adding OT or a CRDT for the description field is a meaningful piece of work — yjs with a y-socket.io adapter would be my pick — and is filed under "what I'd do next."

Reconnection

Socket.io's client reconnects automatically with exponential backoff. I configured infinite retry attempts so a long disconnect (e.g., laptop suspend) recovers gracefully when the network comes back. On reconnect we re-join every project room we were in before, so the user picks up where they left off.

10Security model

Let me walk every layer.

Transport

HTTPS everywhere in production. Render provides a managed certificate; Vercel does the same. The refresh cookie is Secure in production, so it only travels over HTTPS. Helmet adds Strict-Transport-Security.

CORS

Strict allowlist via an explicit origin callback. The frontend Vercel origin is in the list; nothing else is. Credentials are allowed because the refresh cookie needs to cross.

Helmet headers

Default helmet config minus CSP (which needs more careful tuning than fits in this build). That gives me X-Frame-Options, X-Content-Type-Options, Referrer-Policy, X-DNS-Prefetch-Control, HSTS in production, and a handful of less-glamorous protections.

Rate limiting

Per-IP limits on auth (10 / 15 min) and a permissive global limit (300 / 15 min). The auth limit is the brute-force defense. For demo, this is sufficient. In production with real traffic I'd add per-account login limits and a CAPTCHA escalation.

Password handling

bcrypt cost 12, hash stored with select: false so it can't accidentally be returned in a populated query. Passwords are validated for minimum complexity (8+ chars, upper, lower, number) — the brief doesn't dictate the exact rules, and these match common best practice.

JWT secrets

Four separate high-entropy secrets, generated fresh per Render deploy. Never logged. Never serialized to the client. The error handler's stack trace inclusion is dev-only, so a stack trace can't leak a secret in production.

Input validation

Every request body, query, and param is validated by Zod before reaching a controller. ObjectIds are checked against the 24-hex regex. String lengths are bounded. Enums are enforced. The Mongoose schemas are a second layer of defense in case anything slips through.

XSS

The frontend uses React, which auto-escapes all {value} bindings. No dangerouslySetInnerHTML anywhere. The only HTML I generate is in email templates, where the user-supplied fields (name, project name, role) are HTML-escaped by a small helper before interpolation.

CSRF

The refresh cookie is path-scoped to /api/auth and uses SameSite=None only because cross-origin is required. With SameSite=Lax in dev, top-level navigations carry the cookie but most cross-origin requests don't — a partial CSRF defense. For production with SameSite=None, the path scoping limits exposure: only /api/auth/refresh reads it, and that endpoint only ever issues new tokens (it doesn't perform user-facing mutations). The actual mutations (POST /tasks, etc.) require the access token in the Authorization header, which JavaScript on a third-party origin can't read or forge.

For a higher-stakes app I'd add a CSRF token on the refresh endpoint specifically, double-submitted via header. Not required at this assignment's scope.

File upload safety

Mime allowlist on upload (images, common document types, archives, plain text). Size cap at 5MB. The uploaded file's URL is generated server-side from a trusted provider; we never echo back a URL the client could control. Downloads go through a signed-URL redirect with Content-Disposition: attachment for non-images so the browser downloads instead of rendering arbitrary content.

What I deliberately don't do: server-side virus scanning, image dimension validation, server-side image processing. Those are out of scope for the assignment and would be ClamAV / sharp / similar in a real production setup.

Database access

strictQuery: true means queries can't reference fields the schema doesn't know about — small defense against ReDoS-style injection via query parameters. Mongoose's parameterized queries inherently defend against the JS injection equivalent of SQL injection.

Mongo's connection

Atlas requires authentication; the URI carries credentials. The credentials live only in Render's secret store; render.yaml declares sync: false for MONGO_URI meaning it won't be set automatically.

11File upload, end-to-end

Tracing one upload from browser to storage, with every layer.

Step 1: Client picks a file

The task dialog has a hidden file input wrapped in a styled label. Clicking the label opens the OS file picker. On change, the file goes to onFileChange.

Step 2: Client constructs FormData

FormData with two fields: file (the binary) and taskId (string). The data is fed to uploadWithProgress, which uses XHR rather than fetch because fetch doesn't support upload progress.

Step 3: XHR sends multipart

The XHR is opened against /api/projects/:projectId/files with the access token in the Authorization header and withCredentials: true. The upload.onprogress handler updates a state variable that drives the progress bar.

Step 4: Multer parses multipart on the server

The upload route mounts multer's single-file middleware. Multer reads the multipart stream into a Node Buffer in memory (capped at 5MB by config). The buffer attaches to req.file with metadata: originalname, mimetype, size.

Step 5: Controller delegates to file service

The file controller pulls req.file and req.body.taskId, hands off to uploadFile.

Step 6: Service validates

assertAllowed checks the mime type is in the allowlist and the size is within the cap. If the upload claims a taskId, we verify the task exists in the project (defense against a malicious client uploading to a task that's not actually in the project they have access to).

Step 7: Storage adapter uploads

getStorage() returns either the Cloudinary or S3 adapter based on env. Both adapters present the same interface.

The Cloudinary path streams the buffer through upload_stream with the folder, resource type (image vs raw), and unique filename flags set. Cloudinary returns a secure_url and a public_id.

The S3 path constructs a key (folder/uuid-stem.ext), runs PutObjectCommand, and returns a public URL based on the configured endpoint plus the key.

Step 8: Persist the File document

A new File row is inserted with project, optional task, filename, mime, size, URL, public id, resource type, uploader.

Step 9: Attach to task atomically

If there's a taskId, we use findByIdAndUpdate with a $push on the attachments array and an $inc on the version. Atomic; no read-modify-write race.

Step 10: Activity log + socket broadcast

The service calls logActivity with action task.attachment_added. The controller calls emitToProject with task:attachment_added and the file payload. Connected clients update their UI.

Step 11: HTTP 201 to the original caller

The caller (the task dialog) refetches the task to pull in the updated attachments array and re-render.

Downloads

The download endpoint is GET /api/projects/:projectId/files/:fileId/download. The handler loads the file metadata, asks the storage adapter for a signed URL, and 302-redirects to it. The redirect URL is good for 5 minutes; after that the link expires.

For non-images, the signed URL carries Content-Disposition: attachment; filename=... so browsers download instead of rendering arbitrary content. For images we let the browser display them inline (useful for thumbnails).

For production with real S3, you'd use signed download URLs exclusively (which the storage adapter supports). The MinIO public-read policy is a dev convenience.

Mailpit

Catches every SMTP message on port 1025 and shows it in a web UI on 8025. The SMTP auth is permissive (MP_SMTP_AUTH_ACCEPT_ANY: 1) so any credentials work. This means password reset and invitation emails appear in your local browser instead of trying to deliver.

Alternative is MailHog, which is essentially the same idea by a different team. Mailpit is more actively maintained.

What this gives you

A complete local stack. docker compose up -d, then pnpm dev in each project, and SYNC works end-to-end without any external accounts. File uploads land in MinIO and are visible in the console. Password resets land in Mailpit and are clickable.

13Deployment

Two production topologies are supported. (A) Vercel single-deploy is the recommended one — both services on one Vercel app, same origin, simpler cookie story. (B) Vercel + Render is the classic split. Both are first-class; pick the one whose constraints fit you.

For complete copy-pasteable env templates per topology, see docs/ENV.md.

§A. Vercel single-deploy

The repo ships a vercel.json at the root with Vercel's experimentalServices block declaring two services:

{
  "experimentalServices": {
    "frontend": { "root": "frontend", "routePrefix": "/",          "framework": "nextjs"  },
    "backend":  { "root": "backend",  "routePrefix": "/_/backend", "framework": "express" }
  }
}

Vercel routes /, /login, /projects/*, and so on to the Next.js frontend. Anything under /_/backend/* — including /_/backend/socket.io/ — goes to the Express service. From the browser's perspective, the entire app lives on one origin, which simplifies several things in nice ways.

What changes about cookies

On the cross-origin setup, the refresh cookie had to be SameSite=None; Secure with credentials enabled on CORS. On the single-deploy, the cookie can be SameSite=Lax — friendlier defaults, fewer browser warnings, and not vulnerable to the kinds of cross-origin attacks that None opens up.

The subtler thing is the cookie path. Originally I set Path=/api/auth because that was the URL the browser saw. On the Vercel single-deploy, the browser sees auth URLs as /_/backend/api/auth/refresh, so the cookie must be scoped there or it won't be sent back. I made the path configurable via COOKIE_PATH — set it to /_/backend/api/auth on Vercel, leave it at /api/auth on Render. If you forget this, login appears to work but every subsequent refresh fails silently with no cookie attached. Most common gotcha on this topology.

What changes about the Socket.io path

Socket.io's handshake URL is <origin>/socket.io/ by default. On the single-deploy with the backend at /_/backend, the path becomes /_/backend/socket.io/. The frontend's socket client auto-derives this from NEXT_PUBLIC_API_URL: if the URL is absolute (https://api.example.com), it uses the default /socket.io/ path; if the URL is relative (/_/backend), it prepends the prefix. No manual config needed.

What changes about CORS

Same-origin requests bypass the browser's CORS check entirely — the browser sends them without preflight and without enforcing Access-Control-Allow-Origin. The backend's CORS middleware is still configured (defense in depth, in case something later changes the origin model), but in practice it's a no-op for the same-origin path.

What this requires from Vercel

Socket.io needs a long-lived connection. Vercel's experimentalServices with framework: "express" runs the backend as a persistent Node process (not a serverless function), which is what makes WebSocket support work. If at some point you migrate the backend to classic serverless functions, you'd need to move the realtime layer to a managed service (Pusher, Ably, PartyKit). For the assignment surface, the persistent Express service is the right fit.

The actual deploy steps

One: push the repo to GitHub. Two: import on Vercel — it reads vercel.json and picks up both services. Three: under Project Settings → Environment Variables, paste the env block from docs/ENV.md §3.1, splitting between the frontend and backend services as marked. Four: redeploy.

§B. Vercel (frontend) + Render (backend)

If you don't have experimentalServices access yet, or you want the backend isolated from the frontend, the original split works exactly the same as before. The render.yaml blueprint provisions an Express web service. Build is npm install && npm run build, start is npm start, health check at /health.

Env vars fall into three categories. The non-secret defaults (like NODE_ENV=production, TTLs) are set in the blueprint. The JWT secrets are declared with generateValue: true so Render generates strong randoms on first deploy. The actual secrets (Mongo URI, Cloudinary keys, Resend key, CORS origins) are declared with sync: false meaning Render asks for them at deploy time.

Cookies on this topology are SameSite=None; Secure (cross-origin between Vercel and Render). COOKIE_PATH stays at the default /api/auth. COOKIE_SAMESITE=none must be set explicitly.

Frontend setup on Vercel is the same regardless of topology — point the project root at frontend/ and set NEXT_PUBLIC_API_URL to either /_/backend (single-deploy) or the Render URL.

Atlas setup (both topologies)

Create a free M0 cluster, create a database user with readWrite on the database, allowlist 0.0.0.0/0 for trial (in production you'd narrow this to known egress IPs), copy the connection string into the backend env as MONGO_URI.

After deploy: hit /health on the backend (https://your-app.vercel.app/_/backend/health for single-deploy, https://your-api.onrender.com/health for split), open the frontend, register an account, create a project, open the project in two tabs, drag a task between columns in one, watch it move in the other. 14Trade-offs

The 48-hour brief forces choices. Here's where I cut.

No Redis cache

In-memory cache only. Means horizontal scaling immediately costs cache coherence. Decision: defer until there's a reason to scale horizontally. The Cache interface makes the swap trivial.

No background job queue

Emails send synchronously through Resend's HTTP API (which is fast — usually under 200ms). Cloudinary cleanups on file delete are best-effort. A bull/bullmq queue would let me retry failures and offload bursts, but the queue infrastructure itself costs setup time I didn't have.

No Redis-backed Socket.io adapter

One backend instance only. Multi-instance would need @socket.io/redis-adapter for cross-instance message delivery.

No comments, @mentions, task tags, CSV export, analytics, dependencies

All explicitly optional in the brief. Cut to focus on the core.

No rich-text task descriptions

Plain text only. Rich text would imply either a sanitizer pass or a structured format like ProseMirror's JSON. Out of scope.

No fine-grained task permissions

"Member" can edit any task in the project. The brief says "create/edit assigned tasks" but is ambiguous — I read it as "members can create and edit, viewers can only read." A stricter interpretation would require per-task ownership checks. Trivial to add later.

No notification fan-out via socket

Notifications are persisted to the DB and surfaced when the bell icon opens. They're not pushed live to the user's personal socket room. A two-line addition; cut because the demo flow makes the activity feed and toasts sufficient.

No automatic membership-change room eviction

If a user is removed from a project mid-session, their socket stays in the room until they disconnect. Could be solved by tracking each user's sockets and forcing them out on membership changes. Future work.

Limited test coverage

Auth, task CRUD with version conflict, file upload happy-path and rejection. Critical paths only, as the brief allows. No E2E tests; no socket-handler tests.

Plain HTML drag-and-drop

No dnd-kit, no react-beautiful-dnd. The native HTML5 drag API is enough for "drag card between columns." More complex interactions (cross-list reorder with animation) would justify a library.

No dark mode toggle in UI

The CSS tokens support dark mode but there's no toggle in the chrome. Trivial to add — a useState + a class on html + a button.

No telemetry / observability

No Sentry, no OpenTelemetry, no metrics. Pino logs are it.

No image processing

Uploaded images aren't resized or thumbnailed. The frontend renders them at whatever size they were uploaded at, relying on browsers' lazy loading. For production with arbitrary user uploads I'd use sharp or rely on Cloudinary's transformation pipeline.

15What I'd do next

If I had another week, in priority order:

1. Redis everywhere needed

Move the cache to Redis (so we can scale to N backend instances), wire the @socket.io/redis-adapter (so room broadcasts work across instances), and use Redis for rate limiting (so the same per-IP limit applies across instances). One day of careful work.

2. Background queues

BullMQ with a Redis backend. Email sending moves to a queue (Resend already retries internally but a queue gives us our own retry policy and visibility). Cloudinary cleanups on file delete move to a queue. Anything that's currently synchronous-and-best-effort gets durable retry. Half a day.

3. Real-time OT for descriptions

yjs with the y-socket.io adapter. Descriptions become collaboratively editable. The version field on tasks stays for non-description fields; descriptions get OT. One to two days depending on the editor (TipTap on top of ProseMirror would be my pick).

4. Sentry + OpenTelemetry

Error tracking on both frontend and backend. Distributed traces across the HTTP request, socket event, and Mongo query. One day for the basic wiring.

5. Comments with mentions

The brief's biggest cut. A Comment model on tasks, @-mention parsing, notifications fan out. Half a day to a day depending on richness.

6. Tags and CSV export

Tag schema, multi-select on the kanban filter, CSV export endpoint that streams a project's tasks. Half a day.

7. Tighter permissions

Per-task ownership ("member can only edit tasks they created or are assigned to"). A small filter change in the task update controller and the kanban drag handler. An afternoon.

8. Membership-change socket eviction

Track each user's sockets in a Map. On membership removal, iterate the user's sockets and force them out of the project room. Half a day.

9. E2E tests with Playwright

Run two browser contexts, register two users, invite, drag a task in one, assert update in the other. Two days for a meaningful suite covering the realtime flows.

10. CSP and SRI

Content Security Policy on the backend, Subresource Integrity on any third-party scripts (we have none currently, but the moment we add analytics or a font CDN, SRI applies). Half a day with policy reports.

11. Image processing

sharp-based resize on upload, store original + thumbnail variants. Half a day.

12. Audit log surface

The activity feed is project-scoped. An admin-level audit log surface would show every authentication event, role change, and file access. Half a day.
